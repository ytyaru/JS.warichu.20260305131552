# フォールバックの実装とビルド戦略

割注においてフォールバック用アルゴリズムを実装する際の最適な方法論とそのビルド戦略について記す。

## 背景

割注の分割位置算出には複数の戦略（指定位置分割、単語分割、中間分割）が存在する。これらを順次試行し、有効な値が得られなければ次の戦略へ移行する「フォールバック」の仕組みが必要である。しかし、これを割注の実装をする関数内において `for` や `if` などの制御文でベタ書きすると、ビジネスロジック以外のノイズとなり、コードの保守性が著しく低下する。また、最後の戦略は必ず成功するという論理的保証をコード上で表現しようとすると、到達不能な例外処理を記述せざるを得ないという矛盾が生じる。

## 目的

本ドキュメントは、これらの課題を解決するために「制御フローの抽象化」を行い、宣言的な記述を可能にする設計戦略を記したものである。また、抽象化による実行効率の低下というデメリットに対し、ビルドプロセス（AST置換）を活用して「保守性」と「実行性能」を両立させるための戦略についても詳述する。

### 1.1 [割注における分割位置決定方法][]

割注は本文より小さな字で二行に渡り説明を表示する概念だ。これを実現するにあたり、二行に分割する必要がある。この時、分割位置は次のいずれかで決定される。

No|概要|詳細
--|----|----
1|指定位置|`｜`など任意メタ文字によりユーザが分割位置を明示する
2|単語境界|[Intl.Segmenter][]により単語境界で分割する
3|中間位置|割注テキストを等分する

[Intl.Segmenter]:https://developer.mozilla.org/ja/docs/Web/JavaScript/Reference/Global_Objects/Intl/Segmenter

各方法はオプションにて詳細設定できるが割愛する。

ここで重要なのは、1,2,3の順に位置を取得できるか試行し、失敗すれば次の方法を試すことだ。そして最後は必ず3の中間位置で確定する。

このアプローチはフォールバックの概念に似ている。最後の方法だけは確実に実行されると保証されていることが重要だ。

<a id="p1-2-1"></a>
#### 1.2.1. [`if-else if-else`][]

No|概要|API
--|----|----
1|指定位置|`getExplicitSplitIndex`
2|単語境界|`getWordSplitIndex`
3|中間位置|`getDefaultSplitIndex`

```typescript
// 命令型による実装例
for (let i = 0; i < 3; i++) {
  let index: number | null = null;
  let skip = 0;

  // 戦略ごとの実行と合否判定が混在している
  if (i === 0) {
    index = this.getExplicitSplitIndex(graphemes, splitChar);
    skip = 1;
  } else if (i === 1) {
    index = this.getWordSplitIndex(content, totalLen);
  } else {
    index = this.getDefaultSplitIndex(totalLen);
  }

  // 成功した時点で結果を加工して返却する
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}
// 最後の戦略まで失敗した場合の処理（論理的に到達不能）
throw new UnreachableError('すべての分割戦略が失敗しました。');
```

このコードの問題は`if-else if-else`が使われていることだ。もし分割位置の方法が増減したら、`if-else if-else`という文まで書き換えねばならない。これはコードの管理コストが高い。

分割位置が増減する時は、分割メソッドを書き換えるだけで済ませたい。

<a id="p1-2-2"></a>
#### 1.2.2. [関数配列][]

分割位置計算する関数を`strategies`という配列にまとめた。

分割位置計算は数ある戦略の内の一つである。もし戦略が増減したら配列内を書き換えるだけで済む。`for`内のアルゴリズムを書き換えずに済む。

```typescript
const strategies = [
  ()=>this.getExplicitSplitIndex(graphemes, splitChar),
  ()=>this.getWordSplitIndex(content, totalLen),
  ()=>this.getDefaultSplitIndex(totalLen)
];
for (let i = 0; i < strategies.length; i++) {
  const index = strategies[i]();
  
  // 成功判定と中断のロジックがビジネスロジックと混在している
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}
// 最後の戦略まで失敗した場合の処理（論理的に到達不能）
throw new UnreachableError('すべての分割戦略が失敗しました。');
```

このコードの問題は、最終行`UnreachableError`である。ここに到達することは論理上ありえない。フォールバックとは最後の戦略だけは必ず実行されるものだから、それ以降のコードである`UnreachableError`が実行されることはありえない。

では、なぜありえないのにコードとして書いているのか？　それは最後の戦略が`if (index !== null) {`の条件に合致する保証ができないからだ。戦略を変更した時「最後の結果を返す」ということを保証するには、`strategies`の順番と、その中の関数の実装に依存してしまう。

最後の戦略は`getDefaultSplitIndex`であり、これは`null`を返さない。そのように実装している。しかしその実装を変えてしまえば保証できない。また、仮にそれが実装されていても`getDefaultSplitIndex`の順序が最後でなく最後から二番目になれば、保証できなくなる。これは新しく戦略を追加した時に十分発生しうる事態だ。

一々そんなことを考えて`strategies`を調整せねばならないなど面倒極まりない。

<a id="p1-2-3"></a>
#### 1.2.3. [最後の関数だけは確実に実行する][]

そこで、最後の関数だけは条件式を通さず、必ず実行するようにする。

```typescript
const strategies = [
  ()=>this.getExplicitSplitIndex(graphemes, splitChar),
  ()=>this.getWordSplitIndex(content, totalLen),
  ()=>this.getDefaultSplitIndex(totalLen)
];

if (strategies.length === 0) throw new Error("strategiesは一つ以上必要です。");

// 最後の一つ前の戦略まで実行する
const LAST_IDX = strategies.length - 1;
for (let i = 0; i < LAST_IDX; i++) {
  const index = strategies[i]();
  
  // 成功判定と中断のロジックがビジネスロジックと混在している
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}

// 最後の戦略だけ実行する
const index = strategies[LAST_IDX]();
return [
  graphemes.slice(0, index).join(''),
  graphemes.slice(index + skip).join('')
];
```

ポイントは`for`文において最後の一つ前までしか実行しないことだ。そして最後は`for`内`if`を介さずに実行する。これにて最後の戦略だけは絶対に実行するフォールバック方式が実現する。

到達不能なコードという矛盾した記述をせずに済む。

このコードの問題はコードが重複することだ。関数の実行と返却値の作成が重複せぬよう[DRY][]に書きたい。

[DRY]:https://ja.wikipedia.org/wiki/Don%27t_repeat_yourself

<a id="p1-3"></a>
### 1.3 [責任の混在][]

そもそも割注を実装するメソッドの中に、フォールバックを実装するコードを書くことが間違っている。

[ソフトウェア工学][]によれば[関心の分離][]と[単一責任の原則][]に従うべきだ。

No|[ソフトウェア工学][]|概要
--|---------------------|----
1|[関心の分離][]|コードは役割や責任ごとに分割すべきである
2|[単一責任の原則][]|分割した単位は、それぞれ一つの役割・責任だけを持つべきである。

[ソフトウェア工学]:https://ja.wikipedia.org/wiki/%E3%82%BD%E3%83%95%E3%83%88%E3%82%A6%E3%82%A7%E3%82%A2%E5%B7%A5%E5%AD%A6
[関心の分離]:https://ja.wikipedia.org/wiki/%E9%96%A2%E5%BF%83%E3%81%AE%E5%88%86%E9%9B%A2
[単一責任の原則]:https://ja.wikipedia.org/wiki/%E5%8D%98%E4%B8%80%E8%B2%AC%E4%BB%BB%E3%81%AE%E5%8E%9F%E5%89%87

1. 一つの関数やメソッドの中には、一つの役割・責任だけを持ったコードを書くべきである
2. 一つのクラスやモジュールはその上位概念としてメソッドを分類すべきである

一つの関数やメソッドの中に、二つ以上の異なる役割・責任を持たせると、コードは理解不能・修正不能・保守不能になってしまう。

今回、割注の中でフォールバックの仕組みを実装することで、割注の関数は、割注の実装とフォールバックの実装という異なる役割・責任を持った状態になってしまう。これはコードが保守不能になる前兆だ。

#### 2.1.1. [ビジネスロジックの純粋化][]

「分割位置を算出する」という本来の目的（ビジネスロジック）と、「フォールバックの制御」という手段（制御ロジック）が分離される。これにより、ビジネスロジックのコードがノイズから解放され、本来の意図が明確になる。

<a id="p2-1-2"></a>
#### 2.1.2. [変更の局所化][]

フォールバックの仕様変更（例：全失格時の挙動変更）が、割注のビジネスロジックに波及しなくなる。制御ロジックは `fallback` 関数内に閉じ込められているため、修正箇所が明確であり、誤ってビジネスロジックを破壊するリスクが低減する。

<a id="p2-1-3"></a>
#### 2.1.3. [テスト容易性の向上][]

フォールバックの制御ロジックを単体でテスト可能になる。また、ビジネスロジック側は「フォールバックが正しく動くこと」を前提にテストできるため、テストの責務が明確になる。

### 2.2.1. [抽象化による実行オーバーヘッド][]

汎用的な制御フローを抽象化することで、関数呼び出しの回数が増加する。また、戦略を配列や無名関数として渡す構造上、クロージャの生成や動的な関数呼び出しが発生し、直接的な `for` ループや `if` 文によるベタ書きと比較して、実行速度やメモリ消費の面でオーバーヘッドが生じる。

<a id="p2-2-2"></a>
### 2.2.2. [ファイルサイズの増加][]

制御フローを抽象化するためのライブラリコードがバンドルされるため、最終的な出力ファイルのサイズが増加する。特に、小規模なプロジェクトや、極限までファイルサイズを削る必要がある環境においては、このオーバーヘッドが無視できないコストとなる。

<a id="p2-2-3"></a>
### 2.2.3. [学習コストとブラックボックス化][]

独自の抽象化API（`fallback` など）を導入することで、開発者はそのAPIの仕様を学習する必要がある。また、制御フローがライブラリ内部に隠蔽されるため、デバッグ時にスタックトレースが深くなり、問題の発生箇所を直感的に特定することが難しくなる場合がある。

## 3. [API設計][]

<a id="p3-1"></a>
### 3.1 [最終的なAPI構造（fallback関数）][]

割注の分割位置算出において、複数の戦略を順次試行し、有効な値が得られ次第確定する `fallback` 関数を定義する。

```typescript
/**
 * 複数の戦略を順次実行し、条件に合致した最初の結果を返す。
 * 最後の戦略は条件に関わらず必ず実行・返却される。
 */
export function fallback<T, R>(
  strategies: (() => T)[],
  isSuccess: (res: T, index: number) => boolean,
  processor: (res: T, index: number, strategy: () => T) => R
): R {
  const LAST_IDX = strategies.length - 1;
  
  for (let i = 0; i < LAST_IDX; i++) {
    const res = strategies[i]();
    if (isSuccess(res, i)) {
      return processor(res, i, strategies[i]);
    }
  }
  
  return processor(strategies[LAST_IDX](), LAST_IDX, strategies[LAST_IDX]);
}
```
