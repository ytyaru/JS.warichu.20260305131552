# 割注の分割位置算出における論理表現問題とその解決方法

## 背景
割注の分割位置算出には複数の戦略（指定位置分割、単語分割、中間分割）が存在する。これらを順次試行し、有効な値が得られなければ次の戦略へ移行する「フォールバック」の仕組みが必要である。しかし、これを `for` や `if` などの制御文でベタ書きすると、ビジネスロジック以外の「制御ノイズ」が混入し、コードの保守性が著しく低下する。また、最後の戦略は必ず成功するという論理的保証をコード上で表現しようとすると、到達不能な例外処理を記述せざるを得ないという矛盾が生じる。

## 目的
本ドキュメントは、これらの課題を解決するために「制御フローの抽象化」を行い、宣言的な記述を可能にする設計戦略を記したものである。また、抽象化による実行効率の低下というデメリットに対し、ビルドプロセス（AST置換）を活用して「保守性」と「実行性能」を両立させるための戦略についても詳述する。

---

## 1. 背景と課題 (Why)

### 1.1 命令型記述（for/if/return）の限界
ビジネスロジックが制御フローのノイズに埋もれ、可読性と保守性が低下する問題について。

割注の分割位置算出において、複数の戦略を順次試行するロジックを命令型（Imperative）で記述すると、ビジネスロジックが制御フローのノイズに埋もれてしまう。

```typescript
// 命令型による実装例
for (let i = 0; i < 3; i++) {
  let index: number | null = null;
  let skip = 0;

  // 戦略ごとの実行と合否判定が混在している
  if (i === 0) {
    index = this.getExplicitSplitIndex(graphemes, splitChar);
    skip = 1;
  } else if (i === 1) {
    index = this.getWordSplitIndex(content, totalLen);
  } else {
    index = this.getDefaultSplitIndex(totalLen);
  }

  // 成功した時点で結果を加工して返却する
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}
// 最後の戦略まで失敗した場合の処理（論理的に到達不能）
throw new UnreachableError('すべての分割戦略が失敗しました。');
```

ここからコードを改善する。割注の分割位置算出メソッドを`strategies`配列に匿名関数として実装すると、以下のように書ける。

```typescript
// 命令型による実装例
for (let i = 0; i < strategies.length; i++) {
  const index = strategies[i]();
  
  // 成功判定と中断のロジックがビジネスロジックと混在している
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}
// 最後の戦略の実行と、全失格時の挙動がループの外に漏れ出している
```

この記述には以下の問題がある。

1.  **制御ノイズの混入**: 「戦略を順次実行し、合格次第中断する」というフロー制御そのものが、ビジネスロジック（どの戦略を、どのような優先順位で試すか）と密結合している。
2.  **可読性の低下**: 開発者は、コードを読む際に「ループの終了条件」「インデックスの管理」「条件分岐」という、ビジネスロジックとは無関係な実装詳細を脳内でトレースしなければならない。
3.  **保守性の欠如**: 新しい戦略を追加したり、中断条件を変更したりする際、ループ構造全体を修正する必要があり、バグを混入させるリスクが高い。

本来、開発者が記述すべきは「どの戦略をどのような順序で試すか」という**宣言的な定義**であるはずだが、命令型記述ではそれが「手続き」の中に隠蔽されてしまっている。



### 1.2 「最後の砦」の論理的矛盾
言語仕様上、到達不能なパスに対して例外発生コードを強制される矛盾について。

命令型記述の最大の問題は、**「割注の分割ロジック」というビジネスロジックの中に、「フォールバックの制御ロジック」という全く別の責任が混在していること**にある。

本来、割注の分割ロジックは「どの戦略をどのような優先順位で試すか」という**宣言的な定義**に集中すべきである。しかし、命令型記述では、その定義を達成するための「ループの回し方」「中断の判定」「全失格時の挙動」といった**フォールバックの制御ロジック**を、割注のメソッド内に直接記述せざるを得ない。

この責任の混在は、以下の論理的矛盾を引き起こす。

1.  **責任の不当な押し付け**: 割注のメソッドは、本来「分割位置を算出する」という責任のみを負うべきである。しかし、フォールバックの制御ロジックを内包することで、「フォールバックが正しく動作すること」という、本来ライブラリ（あるいは制御フレームワーク）が負うべき責任まで割注が負わされている。
2.  **到達不能パスの強制**: 最後の戦略は論理的に必ず成功する（あるいは強制実行される）ことが設計上自明であるにもかかわらず、命令型記述では「ループを抜けた後」というコードパスが存在してしまう。このため、言語仕様上「到達不能なはずのパス」に対して、開発者は `UnreachableError` のような例外処理を記述せざるを得ない。
3.  **信頼性の低下**: 本来存在しないはずのコードパス（到達不能パス）を記述することは、コードの読み手に対して「ここには到達する可能性があるのではないか？」という疑念を抱かせる。これは、コードの論理的整合性を自ら破壊する行為であり、保守性を著しく低下させる。

### 1.3 責任の密結合問題
戦略、判定、加工、進行管理が1つのメソッドに混在することによる保守性の低下について。

### 2.1 抽象化のメリット
