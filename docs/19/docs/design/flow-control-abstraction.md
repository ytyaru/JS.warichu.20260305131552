# 割注の分割位置算出における論理表現問題とその解決方法

## 背景
割注の分割位置算出には複数の戦略（指定位置分割、単語分割、中間分割）が存在する。これらを順次試行し、有効な値が得られなければ次の戦略へ移行する「フォールバック」の仕組みが必要である。しかし、これを `for` や `if` などの制御文でベタ書きすると、ビジネスロジック以外の「制御ノイズ」が混入し、コードの保守性が著しく低下する。また、最後の戦略は必ず成功するという論理的保証をコード上で表現しようとすると、到達不能な例外処理を記述せざるを得ないという矛盾が生じる。

## 目的
本ドキュメントは、これらの課題を解決するために「制御フローの抽象化」を行い、宣言的な記述を可能にする設計戦略を記したものである。また、抽象化による実行効率の低下というデメリットに対し、ビルドプロセス（AST置換）を活用して「保守性」と「実行性能」を両立させるための戦略についても詳述する。

---

## 1. 背景と課題 (Why)

### 1.1 命令型記述（for/if/return）の限界

割注の分割位置算出は、フォールバック方式を採用する。すなわち複数の戦略（指定位置分割、単語分割、中間分割）を順次試行し、有効な値が得られ次第確定し、得られなければ最後の値で強制確定する。

この要件を[命令型][命令型プログラミング]で記述すると、以下のようになる。

[命令型プログラミング]:https://ja.wikipedia.org/wiki/%E5%91%BD%E4%BB%A4%E5%9E%8B%E3%83%97%E3%83%AD%E3%82%B0%E3%83%A9%E3%83%9F%E3%83%B3%E3%82%B0
[宣言型プログラミング]:https://ja.wikipedia.org/wiki/%E5%AE%A3%E8%A8%80%E5%9E%8B%E3%83%97%E3%83%AD%E3%82%B0%E3%83%A9%E3%83%9F%E3%83%B3%E3%82%B0

```typescript
// 命令型による実装例
for (let i = 0; i < 3; i++) {
  let index: number | null = null;
  let skip = 0;

  // 戦略ごとの実行と合否判定が混在している
  if (i === 0) {
    index = this.getExplicitSplitIndex(graphemes, splitChar);
    skip = 1;
  } else if (i === 1) {
    index = this.getWordSplitIndex(content, totalLen);
  } else {
    index = this.getDefaultSplitIndex(totalLen);
  }

  // 成功した時点で結果を加工して返却する
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}
// 最後の戦略まで失敗した場合の処理（論理的に到達不能）
throw new UnreachableError('すべての分割戦略が失敗しました。');
```

`getDefaultSplitIndex`は必ず`null`以外を返すように実装する。これにより最終行には論理的に絶対到達しないことを期待している。尤も、そのことは実装内容次第であり保証されていない。

ここからコードを改善する。割注の分割位置算出メソッドを`strategies`配列に匿名関数として実装すると、以下のように書ける。`if-else if-else`文を排除する。これにより戦略の数や順序を変更するとき`if`文を修正する必要がなくなる。`strategies`の内容だけを変更すれば良い。

```typescript
const strategies = [
  ()=>this.getExplicitSplitIndex(graphemes, splitChar),
  ()=>this.getWordSplitIndex(content, totalLen),
  ()=>this.getDefaultSplitIndex(totalLen)
];
for (let i = 0; i < strategies.length; i++) {
  const index = strategies[i]();
  
  // 成功判定と中断のロジックがビジネスロジックと混在している
  if (index !== null) {
    return [
      graphemes.slice(0, index).join(''),
      graphemes.slice(index + skip).join('')
    ];
  }
}
// 最後の戦略まで失敗した場合の処理（論理的に到達不能）
throw new UnreachableError('すべての分割戦略が失敗しました。');
```

この記述には以下の問題がある。

1. 責任の混在
2. 保守性の破綻
3. 可読性の欠如
4. ヒューマンエラーの誘発

#### 1.1.1. 責任の混在

ソフトウェア工学では[関心の分離][]と[単一責任の原則][]があるが、上記コードはこれに違反している。

コードは役割や責任ごとに分割すべきである。分割した単位は、それぞれ一つの役割・責任だけを持つべきである。

もしこれを守らなければ理解不能・修正不能なコードとなり破綻するからだ。以降で詳細を話す。

[関心の分離]:https://ja.wikipedia.org/wiki/%E9%96%A2%E5%BF%83%E3%81%AE%E5%88%86%E9%9B%A2
[単一責任の原則]:https://ja.wikipedia.org/wiki/%E5%8D%98%E4%B8%80%E8%B2%AC%E4%BB%BB%E3%81%AE%E5%8E%9F%E5%89%87

#### 1.1.1. 保守性の欠如

コードが修正困難になる。責任区分を定義しないことによって。

本来、割注のメソッドは「分割位置を算出する」というビジネスロジックに責任を持つべきである。しかし、フォールバックの制御ロジック（ループの回し方、中断判定）をメソッド内に混在させているため、割注が「フォールバックのロジック」という本来負うべきでない責任まで負わされている。

この責任の混在により、ビジネスロジックの修正と制御ロジックの修正が同じ場所で発生し、修正すべき箇所とそうでない箇所の区別がつかなくなる。結果として、二つの責任が混在したコード全体を把握しなければ修正できない複雑怪奇で管理不能な状態に陥る。

#### 1.1.2. 可読性の欠如

コードが理解不能になる。責任区分を定義しないことによって。

開発者は、コードを読む際に「ループの終了条件」「インデックスの管理」「条件分岐」という、ビジネスロジックとは無関係な実装詳細を脳内でトレースしなければならない。これにより、ビジネスロジックが制御フローのノイズに埋もれ、本来のビジネスロジックの理解が阻害される。

#### 1.1.3. ヒューマンエラーの誘発

コードに間違いが混入しやすくなる。責任区分を定義しないことによって。

割注のビジネスロジック関数内なのに、フォールバックの実装コードがある。本来触るべきでないフォールバックのロジックまでが編集可能な状態にある。その結果、新しい戦略を追加する際などに、誤って制御ロジックを破壊してしまうリスクが常に存在する。

#### 1.1.4. 結論

責任を混在させることで保守性と可読性が低下し管理不能なコードになる。

本来、開発者が記述すべきは「どの戦略をどのような優先順位で試すか」という[**宣言的な定義**][宣言型プログラミング]であるはずだが、[命令型記述][命令型プログラミング]ではそれが「手続き」の中に混在し、責任が分離されていない。この責任分離の欠如こそが、保守性と可読性を損なう根本的な問題である。



### 1.2 「最後の砦」の論理的矛盾
言語仕様上、到達不能なパスに対して例外発生コードを強制される矛盾について。

命令型記述の最大の問題は、**「割注の分割ロジック」というビジネスロジックの中に、「フォールバックの制御ロジック」という全く別の責任が混在していること**にある。

本来、割注の分割ロジックは「どの戦略をどのような優先順位で試すか」という**宣言的な定義**に集中すべきである。しかし、命令型記述では、その定義を達成するための「ループの回し方」「中断の判定」「全失格時の挙動」といった**フォールバックの制御ロジック**を、割注のメソッド内に直接記述せざるを得ない。

この責任の混在は、以下の論理的矛盾を引き起こす。

1.  **責任の不当な押し付け**: 割注のメソッドは、本来「分割位置を算出する」という責任のみを負うべきである。しかし、フォールバックの制御ロジックを内包することで、「フォールバックが正しく動作すること」という、本来ライブラリ（あるいは制御フレームワーク）が負うべき責任まで割注が負わされている。
2.  **到達不能パスの強制**: 最後の戦略は論理的に必ず成功する（あるいは強制実行される）ことが設計上自明であるにもかかわらず、命令型記述では「ループを抜けた後」というコードパスが存在してしまう。このため、言語仕様上「到達不能なはずのパス」に対して、開発者は `UnreachableError` のような例外処理を記述せざるを得ない。
3.  **信頼性の低下**: 本来存在しないはずのコードパス（到達不能パス）を記述することは、コードの読み手に対して「ここには到達する可能性があるのではないか？」という疑念を抱かせる。これは、コードの論理的整合性を自ら破壊する行為であり、保守性を著しく低下させる。

### 1.3 責任の密結合問題
戦略、判定、加工、進行管理が1つのメソッドに混在することによる保守性の低下について。
