# JavaScript/TypeScript：型安全性の欠如が招く開発の構造的破綻

本ドキュメントは、割注ライブラリの実装過程において直面した、JS/TSという言語仕様が抱える構造的な欠陥と、それが開発者の責任分離をいかに阻害し、開発コストを肥大化させるかを記録したものである。

## 1. JS/TSの言語欠陥による開発コスト激増問題

### 1.1 型はプログラミング言語を成立させる必要条件である
プログラミングにおける「型」とは、メモリ上のビット列をどのように解釈し、どのような操作を許可するかを定義するルールである。例えば、メモリ上の `01000001` というビット列は、「整数」として解釈すれば `65` であり、「文字」として解釈すれば `'A'` である。この解釈方法（型）が確定していなければ、プログラムはデータを正しく読み取ることができない。

品質保証とは、この「解釈のルール」がプログラムの実行中、常に一貫して守られていることを証明するプロセスである。もし実行中に「整数」として扱っていたはずのデータが、ある瞬間に「文字」として解釈されてしまえば、プログラムは意図しない挙動や予期せぬ実行時エラーを起こす。したがって、型を定義し、それを保証することは、プログラミング言語として成立するための最低限の要件である。

### 1.2 JSは型付けできない欠陥言語である

JavaScript はインタプリタ言語でありながら、実行時に型を検証する機構（動的型チェック）が存在しない。型を定義し、それを検証する機構がない以上、プログラムの動作を確定させることは論理的に不可能である。利用時に渡された値の型チェックができないため、予期せぬ実行時エラーや予期せぬ挙動が発生する可能性を排除できない。よってJSは型付け不能により挙動を定義できない欠陥言語である。

### 1.3 TSはJSの欠陥を継承する

TypeScript は実行時に型チェックできない。そのためのコードも生成しない。よってTSはJS同様、実行時に予期せぬ挙動が発生しうる欠陥言語である。

TSはJSに置換(トランスパイル)するツールだ。このとき一度だけ静的型チェックする。しかし実行時に渡される引数値についてはチェック不能だ。それをするには動的型チェックを実装する他ない。だがそのコードをTSが書いてくれることはない。よってTSは動的型チェックできず実行時に予期せぬ挙動が起きうるコードを排出する。

JS/TS共、実行時に型安全を保証できない欠陥言語である。

### 1.4 JS/TSは品質保証不能な欠陥言語である

品質保証とは、プログラムが意図した通りに動作することを保証することである。しかしJavaScriptでは動作を制御できない言語仕様のため保証不能だ。型付けできないからだ。結果、予期せぬ挙動が発生する可能性を常に内包してしまう。

JS/TSは型付け不能という言語仕様により、実行時に渡された値によって予期せぬ挙動が発生する可能性を内包したソフトウェアになる。これは言語仕様上、論理的に解決不能だ。もし自力で動的型チェックを実装しても、それが実行環境において正しく、また網羅された妥当な型チェックであることを保証できない。

型はソフトウェアを構築・実行するコンパイラが責任を持つべきだ。しかし、JSにコンパイラはないし、インタプリタも責任を持たない。そもそも言語仕様により開発者は型付けできない。誰も型について責任を持てない。型はプログラミングを動作させるために必要な情報なのに、誰も責任を持てない仕組みなのだ。よって品質保証など不可能だ。

個々人の開発者が任意で動的型チェック処理を実装することは可能だが、それでは人によって実装によっては抜け漏れや間違いが発生しうる。開発者は言語仕様の欠陥を補うために言語機能を実装せねばならない。それが実行環境の型と矛盾も相違もないことを誰も保証できない。

よってJS/TSは言語仕様上、論理的に品質保証ができない欠陥言語である。

## 2. 保守性・再利用性の喪失

### 2.1 保守性の喪失
JavaScript は型付けができないため、コードの保守は論理的に不可能である。

保守とは、コードの変更が意図した範囲内に収まることを保証する作業である。しかし、型が定義できない言語では、利用時に渡された引数の型が確定せず、コードのどこでデータが破壊されるか予測できない。このため、コードの変更がシステム全体に予期せぬ影響を及ぼすリスクを排除できず、保守という概念自体が成立しない。

#### 2.2 脳内コンパイルの強制

型が定義できない言語では、開発者はコードを読む際に「この値はどの型か」「このメソッドは存在するか」を常に脳内で推論し続けなければならない。これは、本来コンピュータが担うべき型チェックの責任を、開発者が「脳内コンパイル」という形で肩代わりさせられている状態である。この状態では、コードの全体像を把握しなければ修正できず、理解できないから修正もできないという「管理不能な状態」に陥る。

### 2.2 再利用性の喪失

TypeScript はコードをバンドル(結合)する必要がある。内部だけは静的型チェックするためだ。だがバンドルするせいで部品に分けることができなくなり、再利用性が低下する。もし再利用性のあるライブラリにするなら、外部から入力値を受け付ける必要がある。だが動的型チェックができないため、外部から渡された値の型チェックができない。いくら内部の型チェックができていても、大本である外部から渡された値の型チェックができていなければ、実行時に予期せぬエラーが発生し、ライブラリとしての品質保証ができない。

TSの静的型チェックは実行時には無力である。実行前の型チェックしかできない。内部値の静的型チェックという中途半端なもののために、ソフトウェアの再利用性を差し出す価値はあるだろうか？　TSはただJSをバンドルすることで再利用性を喪失するだけの欠陥言語ではないか？

TSを使うことで動的型チェックできないまま、再利用性さえも失う。

### 3. JS/TSの言語欠陥によるビルド・保守コスト激増問題

#### 3.1 インタプリタ言語における最適化の不可能性

JS/TSは実行時の最適化を自動化できない。

コンパイル言語であれば、保守性と可読性の高いソースコードから、実行効率が最大化されたオブジェクトコードを自動生成できる。しかし、JavaScript はインタプリタ言語であり、ソースコードそのものが実行単位である。この仕様上、コンパイルによる最適化（インライン展開やデッドコード削除など）を自動で行うことは不可能である。開発者は、実行性能を追求するために、保守性や可読性を犠牲にした「実行性優先のソースコード」を自ら記述せざるを得ない。

#### 3.2 手動チューニングの必要性

ソースコードは保守性の高い状態を保つべきである。しかし、実行時は実行性の高い状態を保つべきだ。これを保守性の高いソースコード一つだけで実現するのが理想である。コンパイル言語であればそれが可能だった。

JSは不可能だ。JSのようなインタプリタ言語はソースコード自体が実行単位だからだ。故に実行性の高い状態にするためには、保守性・可読性の低いコードを書かねばならない。

#### 3.3 実行性と保守性は両立できない

インタプリタ言語はソースコードが実行単位である。よって実行性の高い状態にするためには、実行性が高く保守性の低いコードを書かねばならない。逆に、保守性を保ったコードであれば実行性が犠牲になる。故に保守性を保ったまま、実行性を保つことは不可能だ。

JSは実行性と保守性は両立できない。

#### 3.4 ソースコード設計負荷

JSはソースコード設計負荷が高い。インタプリタ言語の特性上、実行性と保守性は両立できない。故に開発者は選択を迫られる。すなわち実行性と保守性のどちらを優先するか。あるいは別々のコードを用意するか。

もし保守性を優先すれば実行時パフォーマンスは低下する。
もし実行性を優先すれば保守性が低下し、理解・修正が困難なソースコードになる。

両方のソースコードを用意し、利用時にエンドユーザが選択できるようにすることも可能だ。しかしそれはコードの保守性を著しく低下させる。開発者は同じロジックを異なる構造で二重に実装せねばならない。

コンパイル言語ならばコンパイラが実行してくれた仕事を開発者が手動でせねばならない。この二重管理は、コードの修正時に両方の実装を同期させる必要を生み、同期ミスによるバグの温床にもなる。

決断|問題
----|----
保守性を優先する|実行性が低下する
実行性を優先する|コードの保守性が低下する
両方実装する|プロジェクトの保守性が低下する

JSは型付けできないインタプリタ言語であるため、挙動を制御できず品質保証できない。それを補うために動的型チェック処理を実装しても実行環境と合致することを保証できない。実行性を低下させ、開発コストを増やしてまで実装した挙句、品質保証すら気休め程度。さりとて、動的型チェックの自前実装を放棄すれば、実行時に予期せぬ挙動をする可能性が大量にある危険な状態になる。どちらに転んでも論理的に品質保証できない。ソースコードの二重管理で選択肢を増やした所で、今度は実装負荷や選択負荷が発生する。すべてはコンパイラが自動で最適化すれば解決した話だ。しかしJSにコンパイラはないため、すべて人間がやらねばならない。

#### 3.5 ソースコード実装負荷

* 実行性を優先すれば保守性を担保できる設計から考え実装せねばならず
* 実行性を優先すれば保守性・可読性の低いストレスフルなコードを読み書きせねばならず
* 両方のコードを用意すれば修正時に両方のコードを理解し修正しミスなく同期させねばならない

どれも異常なまでに高負荷だ。

#### 3.6 ソースコード選択負荷

両方コードを用意した場合、エンドユーザはどれを使っていいか分からない。ビルドが最適化してくれたら選ばずに済んだのに。

インタプリタ言語は、エンドユーザにまで負担を強いる欠陥言語である。

### 3.7 48パターンのビルド戦略の必然性

JSはソースコードから実行時最適化ファイルを生成できない。よって開発者が考察せねばならない。すると以下の実装スタイルにより2種類のソースコードを手書きし、それを元にして48種類のソースコードを`bun build`コマンドで生成することになる。

* **ターゲット環境 (3種)**:
    * `bun`: Bunランタイム環境。
    * `node`: Node.js環境。
    * `browser`: ブラウザ環境。
* **モジュール形式 (2種)**:
    * `esm`: ECMAScript Modules。
    * `script`: Classic Script (NodeはCJS、それ以外はIIFE)。
* **実装スタイル (2種)**:
    * `maintenance`: 保守性優先。責任分離されたモジュール構成。
    * `performance`: 実行性優先。責任混在を許容し、最適化されたコード。
* **バンドル戦略 (2種)**:
    * `within`: 共通ライブラリを内包し、単一ファイルで完結させる。
    * `without`: 共通ライブラリを外出しし、再利用性を高める。
* **リリースモード (2種)**:
    * `debug`: デバッグ情報を含み、console系メソッドを維持。
    * `release`: ミニファイ済み、console系メソッドを削除。

これら 3 × 2 × 2 × 2 × 2 ＝ **48パターン** ものビルド成果物を生成する必要がある。これはJSが言語仕様として多数の欠陥を抱えている証だ。

もし他の言語ならばコンパイラで賄えたり、パッケージ管理ツールで賄えたり、そもそも分岐させる必要がなかったりしたものだ。残念ながらJSは欠陥言語であるため、これらが戦略的に価値のあるものとして存在意義を持ってしまう。

ターゲット環境においては実行時のモジュール定義における違いがあるため必要だ。モジュールに関してもESMを使うか否かで実装が変わる。ESMを使えばパフォーマンスが犠牲になるため無視できない。実装スタイルはJS言語の欠陥に基づく実装戦略だ。これもパフォーマンスと保守性のトレードオフになる。バンドル戦略はインポート負荷と再利用性のトレードオフだ。リリースモードは保守性と実行性のトレードオフである。

様々な観点においてトレードオフが発生する。JSは実行速度が遅いため、これらを無視できない。しかし責任分離できない欠陥言語であるため、保守性の喪失も無視できない。結局、言語の欠陥がこうしたビルド負債を生み出した。

### 4. 結論：言語の欠陥により開発＆利用コストが肥大化する

JavaScript/TypeScript は、プログラミング言語として成立するための最低限の要件（実行時の型保証）を満たしていない。

型とはメモリ上のデータ構造やその解釈方法を定義するルールであり、プログラムが意図通りに動作するための前提条件である。しかし、JavaScript には型付けする機能が存在せず、TypeScriptは実行時に型チェックする処理を実装しないため、、実行環境において型安全性を保証する手段は言語仕様のレベルで完全に欠落している。

この欠落により、JavaScript は「意図通りに動作することを保証できない欠陥言語」であるという結論に至る。

この致命的な欠陥を補うために、開発者は「動的型チェックの実装」「ビルドパターンの網羅」「コードの二重管理」という無駄な抵抗を強いられる。しかし、言語仕様の欠陥を、その言語を用いてコーディングし解決することなど不可能である。仮に自力で型チェックを実装したところで、その実装自体にバグがない保証はなく、むしろコードを複雑化させ、新たなバグを作り込む構造になっている。

この「言語の欠陥を補うための無駄な抵抗」こそが、開発コストを肥大化させる真因である。

結論として、JavaScript/TypeScript を選択した時点で、開発者は論理的な品質保証を放棄し、言語の欠陥を補うための「終わりのない手動チューニング」と「肥大化する開発・利用コスト」を支払い続ける運命にある。この言語を選択することは、プログラミングという行為を「論理的な構築」から「言語仕様の欠陥との終わりのない闘争」へと変質させることに他ならない。
