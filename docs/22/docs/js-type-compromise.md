# JS/TS言語欠陥に基づく妥協設計

[JS/TSは欠陥言語である][js-ts-fault.md]。これを前提に、どう実装すべきか考察した。

[js-ts-fault.md]:./js-ts-fault.md

動的型チェック|問題
---------------|----
一切しない|JavaScriptがこれ。実行時に渡された値により予期せぬ挙動が起きる確率が最大化する。
全部やる|実行環境の型を正しく網羅する保証ができない。実装コストが跳ね上がる。保守性が低下する（責任分離不能。言語仕様補完とビジネスロジックが混在する）。パフォーマンスが低下する（読込コード・処理量の増大）。
内部箇所のみやる|TypeScriptがこれ。実行時に改変されないことが確定している箇所だけを一度だけ静的型チェックする。これにより実行時に内部の型を動的チェックせずに済む。だが大本のである外部からの値を受け付ける所で動的型チェックをしないので、結局は実行時に予期せぬ挙動を抑制できない。
外部受付箇所のみやる|内部の型はコード内で保証（正しく利用する箇所以外で呼び出さないこと等）し、テストで証明する必要がある。（引数全型パターン網羅チェックする）

私としては、「全部やる」ことを体系化された言語で実装しつつ、それを「外部受付箇所のみやる」形に落とし込んだJSコードに変換する処理系を実装することを提案する。内部の型についてはTS同様型チェックする。出力結果は型の妥当性を確認できるJSテストコードを出力する。
