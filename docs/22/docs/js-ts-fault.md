# JavaScript/TypeScript における型安全性の欠如と責任分離の不可能性

本ドキュメントは、割注ライブラリの実装過程において直面した、JS/TSという言語仕様が抱える構造的な欠陥と、それが開発者の責任分離をいかに阻害し、開発コストを肥大化させるかを記録したものである。

## 1. JS/TSの言語欠陥による開発コスト激増問題

### 1.1 型はプログラミング言語を成立させる必要条件である

プログラムが意図通りに動作するためには、メモリ上のデータ構造（型）が実行時に一貫して解釈される必要がある。プログラマは、データがどのような型であるかを定義し、そのルールに沿って処理を実装する。この「型の定義」こそが、プログラムの動作を確定させるための必要条件である。もし型が定まらなければ、プログラムは前提が崩壊した状態で実行されることになり、予期せぬ実行時エラーや謎挙動が発生する可能性を排除できない。

### 1.2 JSは型付けできない欠陥言語である

JavaScript はインタプリタ言語でありながら、実行時に型を検証する機構（動的型チェック）が存在しない。これはプログラミング言語として成立するための最低限の要件を満たしていないことを意味する。型を定義できない以上、プログラムの動作を確定させることは論理的に不可能であり、品質保証など語るべくもない。

### 1.3 TSはJSの欠陥を継承する

TypeScript は JS の型安全性を補うために開発されたトランスパイラだが、それは「静的型チェック」というメモ書きをコンパイル時に行うだけであり、実行時には型情報が消滅する。JSへの変換後は型チェックが一切行われないため、TSはJSの「型危険」という欠陥をそのまま継承している。TSは単なるテキスト置換機であり、実行環境における型安全性を一切保証しない。

### 1.4 JS/TSは品質保証不能な欠陥言語である

JS/TSを使用する限り、どれほどテストを重ねようとも、それは「バグの顕在化を遅らせる努力」に過ぎない。言語仕様として実行時の型安全性を保証する機構がない以上、予期せぬ実行時エラーや謎挙動が発生する可能性を論理的に排除できない。これは品質保証が不可能であることを意味する。

## 2. 保守性・再利用性の喪失

### 2.1 保守性の喪失
JSは型危険であるため、そのコードを保守することは論理的に不可能である。定義できないものは保守不能である。利用時に渡された引数で簡単に壊れてしまうからだ。保守もクソもない。まずはこれが最初である。

### 2.2 再利用性の喪失
TSは内部だけは静的型チェックに従ったコードにするためにバンドルする必要がある。だがバンドルするせいで部品に分けることができなくなり、再利用性が低下する。もし再利用性のあるライブラリにするなら、外部から入力値を受け付ける必要がある。だが動的型チェックができないため、外部から渡された値の型チェックができない。いくら内部の型チェックができていても、大本である外部から渡された値の型チェックができていなければすべてご破産だ。

## 3. ビルド戦略によるコードの二重管理

本来であればコンパイラがコードからオブジェクトファイル化し自動で最適化してくれるべき所を、開発者が手動でチューニングせねばならない。

### 3.1 実行性と保守性のトレードオフ
保守性を高めるために責任分離したコードを書くと、実行効率が低下する。実行効率を高めるために責任を混在させたコードを書くと、保守性が低下する。この二律背反を解決するために、同じ処理を複数の異なる方法で実装した別々のコードを生成する必要が出てくる。

### 3.2 肥大化するビルドパターン
言語の欠陥により、実行環境（bun, node, browser）やモジュール形式（esm, iife）ごとに最適化されたコードを用意せねばならない。その結果、ビルドパターンが爆発的に増加し、管理コストが肥大化する。

## 4. 結論：言語の欠陥により開発＆利用コストが肥大化する
JS/TSはプログラミング言語として成立するための最低限の要件を満たしていない。その欠陥を補うために、開発者は「動的型チェックの実装」「ビルドパターンの網羅」「コードの二重管理」という膨大なコストを支払わされている。この言語を選択した時点で、開発者は品質保証を放棄し、肥大化するコストと戦い続ける運命にある。
